# M3 — Implementation notes (read before M4)

State of the codebase after **M3** (`dedup` — the §5.3 perceptual matching engine
+ a **3-stage** review sequence, PLAN.md §8 B, §13 M3). Written for whoever
implements **M4** (trash model: `cleanup` + `trash refresh`, §6).

- **dedup is a 3-stage sequence** (a design revision, folded into PLAN §8 B): one
  `review_runs` row with a `stage` cursor walks **stage 1 `_exact_dup_to_delete\`**
  (byte-identical, default-DELETE) → **stage 2 `_suspect_recompression\`** (tight
  photo band + all video near-dups, default-KEEP) → **stage 3 `_with_minor_edits\`**
  (wider photo band, photo-only, default-KEEP). `--confirm` applies the current
  stage and **auto-advances** to the next non-empty one; after the last it completes.
- **Stage 2 suggests a keep-lead** (photo AND video, annotate-only): the
  least-compressed member's shortcut is marked `_suggested`. Photo ranks resolution →
  lossless-format tier → `detail_score` (zlib size of full-res pixel residual) → size.
  Video ranks resolution → effective-bitrate band → codec-efficiency weight, where
  effective bitrate = `size/duration × codec_weight` (`assets.codec`, schema v5;
  `match.codec_weights` / `match.video_bitrate_tie_pct` config) — this reduces the
  HEVC-master-vs-H.264-export trap. Stage 3 is unranked.
- **Everything works and is tested.** `uv run pytest` → **141 green**. Verified
  end-to-end through the real CLI/daemon (3-stage confirm+advance, resumable
  apply-then-advance window, and the stage-2 `_suggested` lead correctly picking a
  PNG master over a higher-detail_score-but-lossy JPEG sibling).
- **M3 is the first writer** of `similarity_edges`, `review_runs`, `review_actions`,
  and the first to create staging `.lnk`s + the §8.1 audit trail. **Schema at v4** —
  v3 added `review_runs.stage`/`stage_phase` + `review_actions.stage`; v4 added
  `assets.detail_score`. All via an idempotent ADD-COLUMN pass in `init_db`
  (`_migrate_columns`, `_ADDED_COLUMNS` list — there is no migration runner). One
  additive method (`Database.backup_to`).

## 2. The matcher (`matcher.py`) — what M4's `cleanup --perceptual` calls

`cleanup --perceptual` (§6.2) runs the SAME engine, just with a different
comparison set. The two knobs you'll change vs dedup:

- **`load_signatures(db, asset_ids=<folder's active assets>, statuses=('active',))`**
  for the *targets*, and **`statuses=('trashed',)`** for the *pool* — dedup compares
  active-vs-active (trashed excluded, §5); cleanup compares the folder's active
  assets **against the trashed set** (find recompressed copies of trashed content).
- `find_matches(targets, pool, config)` returns canonical-ordered `Edge`s
  (`asset_a < asset_b`, `media_type`, `distance`, `algo`). For cleanup you want, per
  target asset, *which trashed asset it matched* — iterate the edges and pick the
  end that's in the trashed pool (record it as `matched_trashed_asset_id` in
  `review_actions`, a column already in the schema and unused by dedup).

The matcher's photo cutoff is `config.match.t_photo_edit` (32) — the **match
decision**. dedup then **bands** matched photos with `t_photo_recompress` (10) into
its stage 2 (`d ≤ recompress`) vs stage 3 (`recompress < d ≤ edit`); the matcher
itself does no banding. **`cleanup --perceptual` uses `t_photo_edit` alone, no
banding** (one trash-match folder). Video uses the `video.*` knobs + `t_match_video`
(90) and is never banded (single frame-vote). All three cutoffs need real-data
calibration (PLAN §14 #1) — don't trust the output as tuned yet. PDQ blobs are the
32-byte packbits form; `hamming()` is the popcount idiom.

## 3. The dedup job (`jobs/dedup.py`) — shape, because M4 mirrors it

Four modes on one handler, dispatched by params (`_run_dedup`):
`confirm` → `_confirm`, `cancel` → `_cancel`, `dry_run` → `_dry_run`, else `_analyze`.

**Stage planning is per-stage and pure** (no I/O, no writes):
- `_plan_stage(ctx, root_id, root_path, stage)` → `{stage, actions, n_groups, edges}`.
  Stage 1 = `_plan_exact` (external copy → drop all internal, survivor = external;
  else ≥2 internal → keep oldest-mtime). Stages 2/3 = `_plan_perceptual`: run the
  matcher over ALL active assets, then **band** the edges — stage 2 keeps `photo d ≤
  t_photo_recompress` **and every video** edge; stage 3 keeps `recompress < photo d ≤
  edit`. Connected components → groups → one action per member (no survivor).
- **No edge-case-6 exclusion, no deferral** — stage 1 deletes only redundant
  *instances*, so every asset still exists for stages 2/3 in the SAME run.
- `_first_nonempty_stage(start=…)` skips empty stages; used to open on the first
  non-empty stage and to advance. If the whole run is empty → auto-complete "already
  clean" (no dangling pending row).
- **Stage-2 keep-lead** (`_pick_lead` + `_asset_rank_fields`): a group is homogeneous,
  so the group's medium picks the key; winner's shortcut gets `_suggested` + `is_lead`.
  - **Photo:** `(pixels, lossless_format, detail_score, size)` desc as SHIPPED IN M3.
    **⚠ SUPERSEDED — M4 replaced this key with `(pixels, format_rank, size)` and RETIRED
    `detail_score` entirely** (dropped the scan computation AND the `assets.detail_score`
    column, schema v6). Why: detail_score (zlib size of the pixel residual) cost ~40% of
    scan CPU, and JPEG 8×8 blocking inflates it — biasing it toward JPEG both within a
    tier AND across formats — so once banded to tame that noise it only ever agreed with
    file `size` within a format. The current key: resolution → `_photo_format_rank`
    (3-level ordinal lossless > heic/heif/avif > jpg/webp/gif) → file `size` (a clean
    monotonic quality proxy *within* one format; it lies across formats, which the format
    rank handles). See dedup.py `_pick_lead` / PLAN §8 B for the live design.
  - **Video:** `(pixels, bitrate_band, codec_weight)` desc. `effective_bitrate =
    size/duration × codec_weight`; `_log_band` (renamed from `_bitrate_band` in M4, now
    shared) log-buckets it by `video_bitrate_tie_pct` so ~equal bitrates tie → codec
    weight decides. Fixes the common HEVC-master-vs-H.264-export tie the right way;
    genuinely across-codec bitrate error is *reduced not cured* → codec+bitrate shown in
    the manifest for override.
  Both stable-by-path last. Only stage 2 (`suggest_lead = stage == STAGE_RECOMPRESS`);
  stage 3 unranked. **M4 additions:** `_group_lead_and_level` returns *why* each lead won
  (the decision level: resolution / +format / +size), tallied and logged at staging as
  keep-lead pick stats; and `--confirm --keep-suggested` (`_keep_suggested_intended`)
  keeps only each group's `_suggested` lead and deletes the rest, ignoring shortcut edits
  (a no-lead group is spared whole). Annotate-only by default; `--keep-suggested` is the
  opt-in bulk-apply.

**Analyze** computes stage 1, checks the run isn't entirely empty, opens the
`review_runs` row (`stage=<first non-empty>`, `stage_phase='staged'`), then
`_stage_and_pause`.

**`_materialize(stage, actions)`** — **stat-before-create**: only files still on disk
get a `.lnk`; a gone target is lazily forgotten; a **stage-1** target whose survivor
vanished is promoted early (stages 2/3 have no survivor). ⚠ **A shortcut that fails to
create is NOT persisted to `review_actions`** — in a default-KEEP stage confirm reads
an absent shortcut as "delete", so persisting a stage-error would silently delete an
unreviewed file (review-caught bug — keep this). manifest.csv + `proposed_stage{N}.json`
written from staged rows only.

**Confirm (`_confirm`) = apply current stage, then advance:**
1. If `stage_phase != 'applied'`: read shortcut *presence* for the current stage
   (strict filename match; rename == removed; `_intends_delete` inverts by stage
   default), abort if the stage folder is missing (never "delete all"), `backup_to`
   the DB, `_apply_stage` under the authoritative liveness gate (stage 1 keeps exactly
   one survivor, promoting a live target if the recorded survivor vanished; stages 2/3
   flip the asset to `trashed`/`dedup-perceptual` at zero instances), write
   `applied_stage{N}.json`, remove the stage folder, commit `stage_phase='applied'`.
2. `_first_nonempty_stage(stage+1)` → stage it + pause; or `_finalize_completed`
   (write `applied.json`, remove all stage folders, `status='completed'`).

**Resumable apply-then-advance (the one new crash window, §8 B Phase 7):** the two
steps above are separately committed via `stage_phase`. A crash after step 1 leaves
`applied` with nothing staged → re-`--confirm` sees `applied` and jumps straight to
step 2 (does NOT re-delete). Covered by `test_dedup_confirm_resumes_from_applied_phase`.

**Conventions differ by stage** (the bug magnet): stage 1 (`_exact_dup_to_delete`)
present-shortcut = delete / remove-to-spare; stages 2–3 present = keep /
remove-to-delete. `_STAGE_DEFAULT_DELETE` encodes this.

**Owned-root subtlety:** `owned_root` returns the root_id for analyze but **None** for
confirm/cancel/dry-run (they act on the already-owned pending run, or nothing; the
global slot + existing pending row serialize them). Copy this for cleanup's `--perceptual`.

## 4. Reconcile analyze-rollback (`jobs/reconcile.py`) — now live, M4 inherits it

`reconcile_on_startup` now, after flipping stale `running` → `interrupted`, inspects
each interrupted **dedup/cleanup** job's params and rolls back **only a first-stage,
never-confirmed analyze**: `_rollback_analyze` requires the pending run be at `stage=1`
AND `stage_phase != 'applied'` before deleting staging + cancelling. A run **mid-sequence**
(stage > 1, or `stage_phase='applied'`) means an earlier stage's deletions were already
CONFIRMED — it is **left pending** so re-`--confirm` resumes it (the stage cursor picks
up where it stopped). An interrupted `--confirm`/`--cancel` job (by params) is likewise
left pending. It handles `type IN ('dedup','cleanup')`, so M4's `cleanup` gets rollback
for free — register the job type as `cleanup`, use the same params shape
(`{root_id, confirm, cancel, ...}`), and (single-stage) leave `stage=1`.

## 5. Gotchas learned in M3 (don't rediscover these)

1. **`send2trash` rejects `\\?\`** — pass `fsutil.canonicalize()` (plain) form, not
   `extended()`. Everything else (open/scandir/stat/create_shortcut target) still
   uses whichever fsutil form scan uses. See [[dedup-lnk-recyclebin]].
2. **COM per-thread init.** `create_shortcut`/`read_shortcut_target` call
   `CoInitialize`/`CoUninitialize` around each op — safe/refcounted on the worker
   thread. Don't hoist it.
3. **Default-KEEP-stage stage-error = silent delete** (see §3) — never persist a
   review_action whose shortcut isn't on disk (stages 2/3 read absent = delete).
4. **Perceptual deletion is the ONE path an asset becomes `trashed` from a library
   op** (fingerprints retained → future merge excludes it). Stage-1 exact deletions
   keep the survivor so the asset stays `active`. M4's `cleanup` extends the trashed
   set the same way.
5. **`.lnk` staging tests are Windows-only** (`@win_only` in test_dedup.py); the
   validation / dry-run / reconcile tests run everywhere. Mirror this split.
6. **PDQ distance is often 0 for a recompressed near-dup** because both decode to the
   same 512px-downscaled pixels (the downscale normalizes JPEG noise) — not a bug. It
   also means a recompress reliably lands in stage 2 (`d ≤ t_photo_recompress`); to
   land a fixture in stage 3 you need a real pixel edit (see `_photo(tweak=…)`).
7. **Schema columns added to existing tables need `_migrate_columns`** (connection.py)
   — `CREATE IF NOT EXISTS` won't alter a table and there's no migration runner. Add
   an entry to `_ADDED_COLUMNS` (idempotent `ALTER TABLE ADD COLUMN` guarded by
   `PRAGMA table_info`). A live daemon needs `daemon restart` to pick up new code +
   run the migration.
8. **One run spans all 3 stages.** Don't open a new `review_runs` row per stage — the
   partial-unique index is per-root, and `--confirm` advances the `stage` cursor on
   the single row. `status <root>` shows it pending across stages.
9. **`detail_score` was RETIRED in M4 (schema v6) — do not re-introduce it.** M3 shipped a
   residual-entropy `detail_score` (zlib size of the pixel residual) as the photo keep-lead's
   within-tier signal. It was dropped because it cost ~40% of scan CPU (~2s/48MP HEIC) and
   JPEG 8×8 blocking inflates it — so once banded to tame that noise it only ever agreed with
   file `size` within a format. The photo keep-lead is now resolution → `_photo_format_rank`
   (lossless > heic/avif > jpg/webp) → `size`. The scan no longer computes it, the profiler's
   `detail` bucket is gone, and `assets.detail_score` is no longer created on a fresh DB (an
   existing DB keeps a harmless unread dead column — no DROP migration). See PLAN §8 B.

## 7. One-paragraph mental model for M4

The review-run machinery is built and generic (`review.py` + the `review_runs`/
`review_actions` state machine + reconcile rollback + the `matcher`). `cleanup`
(§6.2) is a **second consumer** of it, and simpler than dedup: it is **single-stage**
(`stage=1`, no cursor walk). Default mode is a one-shot exact-hash delete with a
count-confirm (no staging); `--perceptual` is a stateful analyze→confirm run that
reuses the matcher (active-vs-**trashed**, `t_photo_edit` cutoff, **no banding**) and
stages into `_perceptually_identified_trash\` — a **delete-default** folder, same
convention as dedup's stage-1 `_exact_dup_to_delete\`. Before any of that, M4 must
build **"refresh the trash collection"** (§6.1 — index trash-root files → record/flip
assets to `trashed` → empty the folders, record-then-delete ordering) and make
`cleanup`/`merge` call it. You are adding `jobs/cleanup.py`, a trash-refresh routine,
and the `trash refresh` command — you should **not** be touching the matcher, the
review plumbing, the scan pipeline, or dedup's staging.
