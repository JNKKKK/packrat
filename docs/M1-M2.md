# M1 + M2 — Implementation notes (read before M3)

This is the state of the codebase after **M1** (`roots register`/`roots list` /
`scan` / `status`, PLAN.md §8 A + §11) and **M2** (perceptual signatures — PDQ for
photos and video frames written during scan, §5.3). They were built **together**
so scan's "fully fingerprinted" predicate (§8 A2 step 4) is correct from the start
— a plain scan writes hash **and** PDQ in one pass, so M3 slots in with no rescan.

It's written for whoever implements **M3** (`dedup` — the §5 perceptual matching
engine + review-run staging, PLAN.md §8 B, §13 M3).

- **Everything works and is tested.** `uv run pytest` → **110 green**. The
  collection is now populated by exact hash *and* perceptual signature, and each
  completed scan persists a re-viewable report (§3a).
- **The whole §4 schema already existed since M0.** M1/M2 fill `roots`, `assets`,
  `file_instances`, `phash`, `vphash`. M3 is the **first writer** of
  `similarity_edges`, `review_runs`, `review_actions` (all defined, all empty).

## 2. What the collection looks like now (tables M1/M2 populate)

- **`roots`** — `roots register` inserts. `path` canonical + unique; `name` unique
  NOCASE (leaf name or `--name`); `kind` library|trash; `ignore_globs` JSON;
  `last_full_scan_at` stamped **only by `scan --full`**.
- **`assets`** — one row per `content_hash`. `media_type` by **extension**
  (photo|video; RAW→photo). `undecodable=1` + `decode_error` when bytes hash but
  pixels won't decode (§9.1) — such an asset has **no** phash/vphash by design.
- **`file_instances`** — UNIQUE(root_id, path); presence = row existence. `path`
  stored in the **plain canonical form** (no `\\?\` prefix — see §5 gotcha 1).
- **`phash`** — one PDQ row per **photo** asset (`algo='pdq'`, 32-byte `bits`, quality).
- **`vphash`** — one row per sampled **video** frame (frame_index, t_offset_s,
  32-byte pdq_bits, quality). `video.sample_frames` (12) frames at timeline midpoints.
- **`scan_results`** — persisted scan report, **one row per (completed scan job ×
  root)** — accumulates as history (PK `(job_id, root_id)`; keep-all retention).
  Banner counters + flags + `profile_json` (only if `--profile`). Written post-loop
  only for a *completed* scan; dry-run/cancel/interrupt/error write no row. §3a below.
- **`scan_problem_files`** — the undecodable/read-error files behind the counts
  (path + reason), keyed to the scan job. §3a below.

Empty, waiting for M3: `similarity_edges`, `review_runs`, `review_actions`.
(`merge_*` is M5; `embeddings` is M7.)

**PDQ storage format:** `np.packbits(256-bit) → 32 bytes`, stored as a BLOB.
Hamming distance = `popcount(unpackbits(a) XOR unpackbits(b))`. See
`tests/test_media_pdq_downscale.py::_hamming` for the exact idiom M3's matcher wants.

## 3. The scan job (`jobs/scan.py`) — how it's shaped, because M3 reads its output

Scan is the template for a real operation, but it is **not** a simple copy of the
old demo job — it has real internal concurrency. Read its module docstring first.

**Phases (§8 A2):** enumerate (with a per-subtree deletion-detection guard) →
per-file (fast-path skip / hash / resolve against `content_hash` / decode+PDQ /
persist asset+instance+phash/vphash in one txn) → deletion detection → report.

**Two concurrency engines, split by media type** (this is the part that took the
most iteration; details in `[[scan-photo-pipeline]]` memory):
- **Photos → producer/consumer pipeline** (`_run_photo_pipeline`): `smb.io_workers`
  threads read whole files into a **byte-budgeted** queue (`photo_buffer_budget_bytes`,
  1 GB); `smb.cpu_workers` threads hash+decode+PDQ **from the RAM buffer** (pure CPU,
  no fd). Reads each photo **once** and decouples I/O concurrency from CPU concurrency.
- **Videos (and oversized photos) → per-file streamed pool** (`_run_streamed`,
  `smb.scan_workers`): hashing streams in chunks (~1 MiB RAM); video decode seeks to
  ~12 frames and reads only a fraction. **Videos are never buffered** (a 4K clip ×
  workers would blow up RAM).
- **Ordering:** photos run to completion, *then* videos — the two calls are
  sequential and each blocks until its threads join. Not a deliberate barrier, just
  two blocking calls (a common question; it's emergent).

**Fast-path (§8 A2 step 4)** is in-memory: scan preloads all `file_instances` for
the root once, then skips a file whose path+size+tolerant-mtime match **and** whose
asset is "fully fingerprinted." **M3 relevance:** the predicate
(`_asset_fully_fingerprinted`) treats `undecodable=1` as complete (hash-only, no
perceptual — only `--full` retries it) and otherwise requires the media type's
perceptual row to exist. So after M1/M2 every decodable asset has its PDQ; M3 can
trust `phash`/`vphash` are present for `undecodable=0` assets.

**Deletion detection** keys off a **suppressed-subtree** set (errored + ignore-pruned
dirs), not a "clean dirs" set — a genuinely-deleted folder *is* reconciled, an
errored/pruned one is not (§10.1). In-place edits are handled by `_upsert_instance`
forgetting an orphaned prior asset. Both were review-caught bugs; keep the model.

## 3a. Scan-result persistence (`scan_results` / `scan_problem_files`)

Every **completed** scan writes its report to the DB so `status <root>` (and the
M6 TUI) can re-render a past scan — replacing the count-only log banner. A
`ScanReport` collector threads through the workers like the profiler; persistence
happens in `_persist_scan_result` at scan end.

- **One row per (job, root)** — `--all` writes N rows under one job id; re-scanning
  a root **appends** a new row (new job id), so the table is a growing per-root
  **history** (keep-all retention). PK `(job_id, root_id)`.
- **Only completed scans get a row.** Persist runs *after* the root loop, so
  dry-run (gated out), cancel/interrupt (`CancelledError` out of the loop), and
  error (e.g. scanning a trash root) write **nothing**. Resuming an interrupted
  scan = re-run → the resume scan writes the row. `status <root>` shows the latest
  by `job_id DESC` — so after a cancel it shows the last *completed* scan.
- **⚠ Resume-proof gotcha (the important one).** The `undecodable` count + problem
  files are **re-derived from the catalog** at scan end (`assets.undecodable=1`
  JOIN `file_instances` in the root), *not* from what the pass decoded. Reason:
  resume/incremental re-runs fast-path-**skip** undecodables (they're "fully
  fingerprinted"), so a per-pass count would empty out on every re-run. Reading
  committed state instead makes the report describe the *root*, not the pass —
  identical across resume/incremental/`--full`. **read-errors** stay per-pass
  (an unreadable file has no asset to re-derive; it's re-detected each pass).
  This was a real bug caught reviewing the interrupted-scan path — don't regress it.
- **Growth caveat (PLAN §14):** `scan_problem_files` re-inserts a row for *every*
  current undecodable on *every* scan, so it grows per-scan, not per-distinct-
  problem. Fine at small scale; a retention knob is the deferred fix.
- The persist tolerates a closed DB on shutdown (swallows it, like the queue's
  worker writes) so a stop mid-finish can't flip a done scan to `error`.

## 4. The part you'll build most: M3 dedup (mapped to PLAN §8 B)

Dedup is the first **stateful review operation**. Unlike scan it does no file I/O
for matching — it's pure DB/fingerprint math over what scan stored.

### 4a. Register it as a job (owns the target root)
```python
register_job(JobSpec(
    type="dedup", handler=_run_dedup, mutating=True,
    owned_root=lambda p: p["root_id"],   # per-root exclusivity via the queue
))
```
The queue's per-root check (`roots.root_holder`, already used by scan) rejects a
second op on a root with a **pending `review_runs`** row. So opening a `pending`
`review_runs` row at analyze time *is* how dedup "owns" the root until
`--confirm`/`--cancel` — scan's step-1a check (`roots.root_holder`) already refuses
to scan an under-review root. **This machinery exists; you just open the row.**

### 4b. The §5.3 matcher (the real new work)
- Read `phash` (photo) / `vphash` (video) BLOBs; PDQ Hamming in numpy (unpackbits +
  XOR + popcount). Photo: distance ≤ `config.match.t_match_photo` (32). Video:
  duration pre-filter + per-frame ≤ `t_match_video` (90) + `frame_match_fraction`
  vote + `min_comparable_frames`, quality-gated by `min_frame_quality` (§5.3).
- Compare target-folder assets against **active** assets collection-wide (trashed
  excluded). Write canonical-ordered (`asset_a < asset_b`) rows to `similarity_edges`.
- **⚠ PDQ values are downscale-dependent** — see §5 gotcha 2. All hashes M1/M2 wrote
  share the 512px basis, so the matcher is internally consistent; just don't mix
  bases.

### 4c. Staging + state machine (§8 B Phases 2–7)
`review_runs` (pending→completed/cancelled) + `review_actions` (the crash-safe plan)
+ Windows `.lnk` staging under `<root>\_packrat_review\` + the §8.1 audit trail
(`proposed.json`/`applied.json` in `paths.audit_dir()`). `_packrat_review\` is
**already in the ignore set** (`ignore.py REVIEW_DIR`) so scan never indexes it.
The `.lnk` mechanism is unresolved (PLAN §14 #4) — spike `pywin32` early.

### 4d. Reconciliation hook (already stubbed)
`jobs/reconcile.py` has the analyze-rollback comment (delete half-built staging +
mark a `pending` run `cancelled` when its analyze job was interrupted). Still a
no-op — **wire it up in M3** (it's your interrupted-analyze recovery, §3).

## 5. Gotchas learned in M1/M2 (don't rediscover these)

1. **Stored paths are the plain canonical form, never `\\?\`-prefixed.**
   `fsutil.canonicalize()` = the stored/compared form; `fsutil.extended()` adds the
   long-path prefix **only** at the moment of an `open`/`scandir`/`stat`. Store/compare
   with `canonicalize`, do I/O with `extended`. Mixing them broke path equality (a
   fixed bug — root-level files got the prefix, nested didn't).
2. **PDQ hashes are downscale-edge-dependent** (`match.pdq_max_edge=512`, LANCZOS
   before PDQ — a ~7× speedup, drift <8/256). Hashes computed at one edge aren't
   bit-identical to another; changing the knob needs a `scan --full` re-PDQ. See
   `[[pdq-downscale]]`. Dimensions stored are **full-res** (only the PDQ input shrinks).
3. **Capture time comes from PIL/PyAV inline, not a per-file exiftool subprocess** —
   a subprocess per file is ruinous over SMB. See `[[scan-metadata-no-exiftool]]`.
   Synthetic test fixtures carry no EXIF, so `captured_at` is NULL for them.
4. **Profiler numbers:** bucket seconds are **thread-summed** (can exceed wall — that's
   the parallelism signal); `MB/s` is **aggregate = bytes / wall**, not per-thread.
   See `[[profiler-metrics]]`. Real calibration: gigabit SMB NAS saturates ~87 MB/s;
   scan is I/O-bound after the PDQ fix. First full scan is a one-time cost (§5.4).
5. **`scan --full` re-hashes but does NOT re-decode** a hit that's already
   fully-fingerprinted (§8 A2 step 6) — so `decode: 0.0s` on a full rescan of
   unchanged content is correct, not a bug.
6. **Daemon log rotates at midnight** into `logs/daemon.log.YYYY-MM-DD`; raw
   fds go to `logs/daemon-bootstrap.log`. Don't point the fd redirect back at
   `daemon.log` and keep uvicorn's `log_config=None`. See `[[daemon-log-rotation]]`.
7. **Fixed port 51789, one daemon per machine** (unchanged from M0). The live daemon
   runs old code until `packrat daemon restart`. Isolated-home tests collide with a
   live daemon — use the in-process `TestClient` (see `tests/test_api*.py`).
8. **`from __future__ import annotations` + FastAPI:** request-body Pydantic models
   must be **module-level** (`SubmitJobRequest`, `RegisterRootRequest` in server.py).

  just logs that it's deferred).

## 7. Config knobs M3 uses (§9.2) — all in `config.py` + shipped `DEFAULT_CONFIG_TOML`

- `match.t_match_photo` (32), `match.t_match_video` (90) — the §5.3 cutoffs. **Both
  need calibration on real data before trusting dedup output** (PLAN §14 #1).
- `match.pdq_max_edge` (512) — the downscale edge (see gotcha 2).
- `video.*` — sample_frames/duration_tol/frame_match_fraction/min_frame_quality/
  min_comparable_frames — the §5.3 video-match structure knobs.
- `review.low_quality_hint` (50) — the annotate-only photo-quality confidence flag.
- `audit.retention_days` (0=keep forever) — the §8.1 audit trail knob.

Config is a **frozen snapshot per job start** (`ctx.config`); don't re-read inside a job.

  skips rather than errors.

## 9. One-paragraph mental model for M3

Scan (M1/M2) has filled the catalog with exact-hash identity **and** perceptual
signatures (photo PDQ / video per-frame PDQ), all at a consistent 512px PDQ basis.
M3 `dedup` is the first operation to **read** those signatures: a new job type that,
at analyze time, opens a `pending` `review_runs` row (which owns the root via the
existing per-root exclusivity machinery), runs the §5.3 Hamming matcher over
`phash`/`vphash` to populate `similarity_edges`, writes a crash-safe `review_actions`
plan, stages `.lnk`s under `_packrat_review\`, and pauses — then `--confirm` reads
shortcut presence and deletes to the Recycle Bin. You are adding `jobs/dedup.py`, the
matcher, the `.lnk` staging, and wiring the reconcile.py analyze-rollback hook — you
should **not** be touching the scan pipeline, the queue, or the schema.
