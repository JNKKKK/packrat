# M5 ‚Äî Implementation notes (read before M6)

State of the codebase after **M5** (`merge` ‚Äî the ¬ß8 C exact-hash
"discard trash + copy what's new" workflow). Written for whoever
implements **M6** (the TUI ‚Äî ¬ß12, ¬ß13 M6).

- **M5 makes `merge_runs`/`merge_plan_items` real.** Both tables were
  defined + empty since M0; merge is their first (and only) writer. No
  schema change ‚Äî **schema stays at v7**.
- **Everything works and is tested.** `uv run pytest` ‚Üí **241 green** (was
  237 after the M5 runtime bits landed; **+55 vs M4's 182**). New:
  `test_merge.py` (19), `test_api_merge.py` (8), +1 merge label test, +4
  `resolve_dest`/`ignore_merge` tests in `test_roots.py`. Merge's copy path is
  plain `shutil.copyfile` + atomic rename, so ‚Äî unlike dedup/cleanup/trash which
  recycle files ‚Äî it needs **no Windows shell APIs** and its tests run
  everywhere (no `@win_only`). Verified end-to-end through the real
  CLI/daemon (register ‚Üí scan ‚Üí dry-run ‚Üí merge ‚Üí status ‚Üí idempotent re-run,
  structure-mirrored, source untouched) and via the in-process TestClient

## 2. The merge job (`jobs/merge.py`) ‚Äî shape

`merge = discard trash + copy what's new`, decided **entirely by exact content
hash** ‚Äî no matcher, no `similarity_edges`, no review folder, no interactive
pause (the opposite of dedup/cleanup). Dispatch: `dry_run` ‚Üí `_dry_run`, else
`_merge`.

**`_merge` (Phases 0‚Äì4):**
1. `_resolve_and_validate` ‚Äî re-read the dest root (daemon already resolved
   `--into` ‚Üí `root_id` + `dest_path` in params); validate the transient source
   (exists / dir / non-empty / readable) and source‚áÑdest disjointness.
2. `trash.refresh_trash(ctx)` ‚Äî **runs for real even under `--dry-run`** (¬ß6.1).
   The trashed set must be current before classification.
3. Find an **open** `merge_runs` row for this dest (`planning`/`copying`) ‚Üí resume
   it (prints "resuming interrupted merge from ‚Ä¶: N of M done"); else open a fresh
   `planning` row.
4. Unless already `copying`: opportunistic fast-path `_scan_dest` (best-effort;
   failure only makes the compare set stale, never wrong) ‚Üí `_build_plan`.
5. `_backup_db` ‚Üí `_copy_and_register` ‚Üí mark `merge_runs.status='done'` ‚Üí report.

**`_build_plan` (Phase 1 + 2):** enumerates the run's **frozen `source_path`**
(not the freshly-passed arg ‚Äî the plan owns source+dest), BLAKE3s each file into
`merge_plan_items(progress='pending')`, collapses exact-within-source dups
(oldest-mtime rep, rest `dup-in-source`), classifies each rep
(`_classify_hash`: miss‚Üí`new`, trashed asset‚Üí`trashed`, active asset‚Üí`exact-known`),
then flips `status='copying'` in **one transaction** ‚Üí the plan is now **frozen**.

**`_copy_and_register` (Phase 3), resume-aware per `merge_plan_items.progress`:**
- `error`/no-hash ‚Üí count error.
- non-`new` (`exact-known`/`trashed`/`dup-in-source`) ‚Üí count, mark `skipped`.
- `new` + already `registered`/`copied-unindexed` ‚Üí count without touching disk.
- `new` + `copied` (crash between rename and DB write) ‚Üí skip re-copy, just register.
- `new` + `pending` ‚Üí `_place_copy` (mirror structure, collision ‚Üí hash-compare ‚Üí
  reuse-in-place or `(1)` rename; temp-write ‚Üí **verify BLAKE3 == frozen hash** ‚Üí
  atomic `os.replace`; set `progress='copied'`+`dest_path`) then `_register_or_mark`.

**`_register_or_mark` (Phase 3 step 11 ‚Äî the silent-forget fix):** tests the
copied file's path **relative to the root** against the dest root's ignore set via
**`_dest_ignored`**, which reproduces *exactly what a later scan would enumerate* ‚Äî
allowlist + file globs **+ ancestor-directory pruning** (`is_dir_pruned`, so a
`Screenshots/` or `**/cache/**` rule excludes the whole subtree). Ignored ‚Üí
`progress='copied-unindexed'` (copied to disk, NOT catalogued). Else ‚Üí insert an
`active`, **un-perceptual** asset (`probe_metadata` fills display cols, phash/vphash
left for a later scan) + `_upsert_instance`, `progress='registered'`. Idempotent
(asset by `content_hash`, instance by `(root_id, path)`) so a resume replay is safe.

**`_dry_run`:** Phases 1‚Äì2 **in memory** (no `merge_runs`/`merge_plan_items`, so it
neither trips the cross-op guard nor leaves a resumable run), skips `_scan_dest`
(mustn't mutate the catalog), computes the would-be-ignored dests up front ‚Äî **but
`refresh_trash` still runs for real** (¬ß6.1).

## 3. Cross-op guard & resume (the subtle bits ‚Äî copy for M6's queue view)

- **Merge OWNS the dest root** (¬ß8 C Phase 0 step 2a). `owned_root = None if
  dry_run else root_id`. The dequeue gate holds a fresh merge behind a pending
  review / another open merge; a `--dry-run` merge owns None (writes nothing).
- **`ignore_merge_holder=True`** is what lets a *resuming* merge run: without it,
  its own open `merge_runs` row would read as a holder and the job would be held
  forever waiting on itself. The gate still honors a **pending review** (only the
  merge-run check is skipped). Mirrored in `status`/queue "blocked" reasons via
  `queries._annotate_queued_row` (threads `spec.ignore_merge_holder`).
- **Resume trusts the frozen plan** (¬ß8 C): a `copying` run replays verbatim (no
  re-hash, no re-classify ‚Äî the accepted cost is a possible redundant byte-dup if
  the collection gained a matching asset in the crash‚Üíresume window; `dedup`
  collapses it later). A `planning` run (crash before the plan froze) wipes partial
  Phase-1 items and rebuilds. The `copied‚Üíregistered` gap is closed by per-item
  `progress`.
- **Reconcile (M0) needs no merge changes.** `_is_destructive_apply` returns False
  for `merge` (copy-only), so an interrupted/queued merge **drains automatically** ‚Äî
  and re-running auto-resumes the open run. Verified: an interrupted merge is left
  `interrupted` by reconcile; the *open `merge_runs` row* is what resume keys off,
  not the job status.

## 4. Gotchas learned in M5 (don't rediscover these)

1. **`refresh_trash` runs for real under `merge --dry-run`** (¬ß6.1, same as M4
   cleanup). Only the *copy* + plan/asset writes are skipped.
2. **A merge-created asset is `undecodable=0` with NO phash** ‚Äî *not-yet-fingerprinted*,
   NOT undecodable (¬ß4). This is exactly the case scan's step-6 **backfill exception (a)**
   fills in on the next `scan <dest>` (the fast-path won't skip it ‚Äî it fails the
   "fully fingerprinted" predicate). `probe_metadata` deliberately never sets
   `undecodable` even on a probe failure ‚Äî it just leaves metadata NULL; the backfill
   scan sorts out real decodability.
3. **`_dest_ignored` must mirror scan's *directory pruning*, not just file globs.**
   A `Screenshots/` (dir-only) glob is applied by scan's walker via `is_dir_pruned`,
   never per-file ‚Äî so merge must walk the copied file's **ancestor dir segments** and
   test each with `is_dir_pruned`, or an ignored-subtree file gets wrongly registered
   (then silently forgotten by the next scan ‚Äî the very bug ¬ß8 C step 11 fixes).
4. **`resolve_dest` is path-first, then name** (like `resolve_root`, ¬ß11) ‚Äî a root
   handle must not shadow a real path arg. It also matches by **containment**
   (`is_within`), unlike `resolve_root`'s exact/name match, because `--into` may be a
   subfolder of a root.
5. **Cooperative cancel does NOT flip `merge_runs='cancelled'`** despite ¬ß8 C's prose.
   The runtime shares ONE cancel flag between a user-cancel and a clean `daemon stop`,
   and ¬ß3 requires a clean stop to stay **resumable** ‚Äî so merge favors "a stop never
   loses in-flight progress" and leaves the run open (copy-only ‚áí partial copy is
   safe; re-run resumes). A true user-cancel that discards the plan would need the
   runtime to distinguish the two signals ‚Äî a ¬ß3/M0 change, out of M5's scope.
   Documented in `merge.py`'s module docstring.
6. **`_copy_and_register` copies to the run's FROZEN `dest_path`**, and `_build_plan`
   enumerates the run's FROZEN `source_path` ‚Äî both read from `merge_runs`, not the
   freshly-passed CLI args, so a resume can't drift to a different `--source`/`--into`
   pointing at the same dest root.

## 5. CLI surface now

```
packrat merge <source> --into <dest> [--dry-run] [--detach] [--json]
```
`--into` is a root **name** or a **path** (a subfolder of a library root is fine ‚Äî
created at copy time). Source is read-only; dest is copy-only. One shot, resumable
from its plan on crash (no `--confirm`/`--cancel` ‚Äî those are dedup/cleanup verbs).
Follow with `scan <dest>` + `dedup <dest>` to fingerprint the copied files and clean
recompressed near-dups merge let through (exact-hash-only ‚Üí a recompressed copy of
trashed/owned content copies as `new`; ¬ß14 #6).

`result_json` shape (the ¬ß4/¬ß12 uniform outcome ‚Äî M6 renders this):
```json
{"op":"merge","dry_run":false,"source":"‚Ä¶","dest_root":"iPhone",
 "new":N,"exact_known":X,"trashed":Z,"dup_in_source":W,"collisions":R,
 "unindexed":U,"errors":E,"summary":"N copied ¬∑ X known ¬∑ Z trashed ¬∑ W dup-in-source"}
```
(`unindexed` = files copied to an ignored dest path, left uncatalogued ‚Äî ¬ß8 C step 13.)

## 6. One-paragraph mental model for M6 (TUI, ¬ß12)

The daemon runtime is now **feature-complete for v1** (M0‚ÄìM5): register/scan/dedup/
merge/cleanup/trash/untrash all submit `jobs` rows, stream SSE progress, write a
uniform `jobs.result_json`, and expose per-root history via `jobs.root_id` +
`scan_results`/`merge_runs`/`review_runs`. **M6 is a pure presentation layer** ‚Äî a
Textual app (`packrat` no-args) that subscribes to the running job's SSE stream and
polls the read-only snapshots (`queries.status_snapshot` / `root_detail` /
`recent_jobs` / `root_jobs` / `queued_jobs`) already used by `packrat status`/`jobs`.
Everything the TUI needs exists: the **global Queue** (`queued_jobs` + `blocked`
reasons, running job), the **per-root jobs** list (`root_jobs`), the **job
detail/result card** (`job_detail` ‚Üí `result_json` + the per-op tables), and job
labels (`jobs.job_label`, already covers every type incl. merge). You should **not**
need to touch any job handler, the queue, or the matcher ‚Äî only add the Textual
front-end (and the two deferred `roots unregister`/`rename` verbs, ¬ß14 #9, if the
"Manage roots" panel needs more than add+list). See ¬ß12 for the layout.
