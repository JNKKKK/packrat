# M4 — Implementation notes (read before M5)

State of the codebase after **M4** (the trash model: "refresh the trash
collection" §6.1, `cleanup` §6.2, `untrash` §6.3). Written for whoever
implements **M5** (`merge` — §8 C, §13 M5).

- **M4 makes the trashed-hash set real.** Before M4 an asset only became
  `trashed` via dedup's perceptual discard (§8 B). Now three more paths do:
  `trash refresh` (absorb a trash-folder file → new/flipped trashed asset),
  `cleanup --perceptual --confirm` (a confirmed near-dup trash match), and the
  reverse — `untrash` — which *forgets* a trashed fingerprint. **M5 merge reads
  this set** (exact-hash trash exclusion, §8 C Phase 2).
- **Everything works and is tested.** `uv run pytest` → **182 green** (was 141
  after M3). New: `test_trash.py`, `test_cleanup.py`, `test_untrash.py`,
  `test_api_cleanup.py`, +1 dedup status-count test. Windows `.lnk` staging /
  Recycle-Bin deletion is exercised for real (this box is win32); the DB-only
  half (recording, validation, dry-run, reconcile) runs everywhere.
- **No schema change.** M4 uses columns M3 already added — `review_runs`
  (run_type `'cleanup-perceptual'` was always in the CHECK), `review_actions`
  (`matched_trashed_asset_id` was defined-but-unused until now; `stage=1` for
  single-stage cleanup). Schema stays at **v5**.

## 2. The shared refresh procedure (`trash.py`) — **M5 reuses this**

`refresh_trash(ctx) -> summary_dict`. Merge (§8 C Phase 0 step 3) and cleanup
both call it at the start so the trashed set is current before they compare.

- **Enumerates every `kind='trash'` root** with the same `IgnoreSet` +
  `scan.enumerate_root` scan uses (one scandir round-trip per dir, §10.1). An
  **offline** trash root is skipped + logged (not read as "emptied", §10.1).
- **Per file: hash always; decode only on a content-hash MISS.** A hit already
  has its signatures from a prior scan/refresh, so re-decoding would waste SMB
  I/O. New content → `_create_trashed_asset` (idempotent on `content_hash`, +
  `_insert_perceptual` reused from `jobs/scan`); an **active** match flips to
  `trashed` (retain fingerprints); a **trashed** match is a no-op.
- **RECORD-THEN-DELETE (§6.1, required):** the fingerprint commits **before**
  the file is recycled. A crash between just re-processes survivors next run
  (recording is idempotent). An **undeletable** file (locked) is left in place +
  reported — never blocks the whole refresh.
- **No dry-run.** `refresh_trash` always runs for real, even when its caller is
  `--dry-run` (§6.1's always-absorb rule). Merge/cleanup skip only their *own*
  destructive step under dry-run; refresh still empties the trash inboxes. **M5
  must call `refresh_trash` for real even in `merge --dry-run`.**

`trash.py` imports `jobs.scan` (for `enumerate_root`/`Candidate`/
`_insert_perceptual`) — that's the only in-package coupling; no cycle (scan
doesn't import trash).

## 3. The cleanup job (`jobs/cleanup.py`) — shape

One handler, dispatched by params (`_run_cleanup`):
`cancel`→`_cancel`, `confirm`→`_confirm`, `apply`→`_apply_default_exact`,
`perceptual`(no confirm/cancel/dry_run)→`_analyze_perceptual`, else→`_preview`.

**Default exact mode is CLI-orchestrated across TWO jobs** (the daemon is
headless, so the typed count-confirm can't live in the job):
1. CLI submits a **preview** job (no flags) → `_preview`: refresh trash + count
   exact-trash matches, act on nothing.
2. CLI reads the count via **`GET /cleanup/preview`** (read-only
   `queries.cleanup_exact_preview`), prompts `Type the count (N)`, warns about
   network-share files (permanent, §10).
3. On confirm the CLI submits an **apply** job (`apply=True`) →
   `_apply_default_exact`: DB backup, then recycle each exact match.
   *(Accepted: the count-confirm is not atomic with the delete — another op could
   change the trashed set in the gap. Apply re-queries committed state + re-checks
   the holder, so it never deletes more than currently matches. If M5 wants an
   atomic variant, that's a design change, not a bug.)*

**`--perceptual` is a single-stage review run** (mirrors dedup but `stage=1`, no
cursor walk):
- `_analyze_perceptual`: refresh → open a `pending` `review_runs` row
  (`run_type='cleanup-perceptual'`) → record **exact** matches in the plan (NOT
  deleted yet) → run the **matcher active-vs-trashed** (`_perceptual_candidates`)
  → stage perceptual candidates as `.lnk` under
  `_packrat_review\_perceptually_identified_trash\` (**delete-default**: a staged
  shortcut = will delete; remove to spare) → `proposed.json` → pause.
- `_confirm`: Phase-5 guard (staging folder must exist if any perceptual staged),
  read shortcut *presence* (present → delete), DB backup, delete exact +
  still-staged perceptual together, `applied.json`, mark `completed`. A confirmed
  perceptual delete flips its asset to `trashed`/`cleanup-perceptual` at zero
  instances (fingerprints retained — future merge excludes it).
- `_cancel`: discard staging, mark `cancelled`, delete nothing.

**Matcher usage (the M3 engine, verbatim):**
`load_signatures(db, asset_ids=<folder active>, statuses=('active',))` for
targets, `statuses=('trashed',)` for the pool. `find_matches` → pick the trashed
end of each edge → `matched_trashed_asset_id`. **Single wider `t_photo_edit`
cutoff, NO recompress/edit banding** (that stage split is dedup-only).

**owned_root subtlety (copy for merge's cross-op guard):** only the
`--perceptual` **analyze** owns the root (opens the pending run → queue rejects a
second op). Preview / default-apply / confirm / cancel / dry-run own **None**;
preview + apply re-check the holder in-handler via `_reject_if_held`
(`roots.root_holder`), confirm/cancel act on the already-owned pending run. The
global slot serializes them.

**Reconcile is FREE (M3 already handles `type='cleanup'`).** An interrupted
cleanup *analyze* (stage 1, not confirm/cancel) rolls back (delete staging +
cancel run); an interrupted `--confirm` stays `pending` for an idempotent re-run.
Verified by `test_reconcile_*_cleanup_*`.

## 4. untrash (`jobs/untrash.py`) — the reverse

Hashes the presented file(s) (a file, or a folder walked with the scan ignore
set — **NOT** resolved against roots, §6.3), matches by **exact content hash**:
- trashed asset **with ≥1 live instance** → flip back to `active` in place.
- trashed asset **with zero instances** → forget entirely (delete asset, cascade
  fingerprints) → content treated as brand-new if it reappears.
- active/unknown → no-op. **Never creates an asset.**

Owns no root, does NOT call refresh, so `--dry-run` truly changes nothing.

## 5. Gotchas learned in M4 (don't rediscover these)

1. **`refresh_trash` runs for real under any `--dry-run`** (§6.1). M5 merge's
   `--dry-run` must still call it — dry-run scopes only merge's *copy* step, not
   the trash absorb. There is intentionally NO dry-run refresh.
2. **RECORD-THEN-DELETE is load-bearing.** Never recycle a trash file before its
   fingerprint commits, or a crash loses the trashed memory. `_absorb_file`
   returns True only on a committed record; only then does `_empty_file` run.
3. **`matched_trashed_asset_id`** (schema since M0, unused until M4) is how a
   cleanup perceptual action records *which* trashed asset it matched. dedup
   leaves it NULL. M5 doesn't touch it.
4. **`status` counts are stage-scoped** (`_review_counts(conn, id, type, stage)`).
   A dedup run keeps confirmed earlier-stage `review_actions` rows, so an
   unscoped count over-reports "to delete". Cleanup is `stage=1`, so the filter
   is a no-op there — but keep passing the stage.
5. **Default cleanup = two jobs (preview → apply).** The typed count-confirm is
   in the CLI, not the job (headless daemon). `/cleanup/preview` is the read-only
   count surface. Merge has NO interactive pause, so it won't need this pattern —
   but note the same "refresh commits before the guarded step" shape.
6. **A trashed asset can legitimately keep instances.** After `trash refresh`
   flips an *active* library asset to `trashed` (a byte-dup was dropped in a
   trash folder), the library file stays on disk until a `cleanup` removes it —
   so a `trashed` asset with ≥1 instance is normal now, not just the zero-instance
   trash-memory case. `untrash` reactivates exactly that in-place case.

## 7. One-paragraph mental model for M5 (merge, §8 C)

Merge is `discard trash + copy what's new`, decided **entirely by exact
content hash** — no matcher, no review folder, no interactive pause (the
opposite of cleanup/dedup). At Phase 0 it **calls `trash.refresh_trash(ctx)`**
(the M4 routine, for real even under `--dry-run`), takes the dest-root cross-op
guard (an open `merge_runs` row — the `owned_root`/`root_holder` machinery
already checks it, §4/§8 C Phase 0), then fingerprints the *transient source*
(BLAKE3 only, persisted to `merge_plan_items` for resume), classifies each rep
by exact hash against the DB (`dup-in-source`/`trashed`/`exact-known`/`new`),
copies the `new` files (hash-verify → atomic rename, structure-mirrored) and
registers them as un-perceptual assets (a later `scan`/`dedup` backfills
phash/vphash). You are adding `jobs/merge.py` + the `/merge` endpoint + `packrat
merge`; you should **not** touch the matcher, review plumbing, cleanup, or
`trash.py` (you call it). `merge_runs`/`merge_plan_items` are defined + empty
since M0; merge is their first writer.
