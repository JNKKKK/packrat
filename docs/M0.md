# M0 — Implementation notes (read before M1)

This is the state of the codebase after **M0** (skeleton + job runtime + decode
smoke test, PLAN.md §13). It's written for whoever implements **M1** (`roots
register` / `roots list` / `scan` / `status`, PLAN.md §8 A + §11).

M0 built the **runtime that operations plug into** — not any operation itself.
There is no `scan`/`dedup`/`merge` yet. Your job in M1 is to add the first real
operation (`scan`) as a **job type** on top of this runtime, and the first
catalog writes.

- **Everything works and is tested.** `uv run pytest` → all green. The daemon
  auto-spawns, runs jobs in the background (headless on Windows), streams live
  progress, cancels cooperatively, and recovers from crashes.
- **The whole §4 schema already exists** (all 13 tables). M1 mostly *fills* tables
  scan is responsible for; you should not need schema changes.

## 2. Runtime architecture (how a job runs)

Two thin clients (CLI now, TUI in M6) drive a background **daemon** that owns the
DB and a **single-worker job queue**. This is exactly PLAN.md §3; here's the M0
realization:

```
packrat scan D:\… (Typer, cli/main.py)
   │  ensure_daemon()  ── bind-or-connect, auto-spawn if down (spawn.py)
   ▼
DaemonClient (client.py)  ──HTTP+token──►  FastAPI (server.py)
                                              │  queue.submit("scan", params)
                                              ▼
                                        JobQueue (queue.py)
                                              │  one mutating job at a time
                                              ▼  worker thread
                                        JobSpec.handler(JobContext)   ← your scan code
                                              │  ctx.progress(...)  → SSE → client bar
                                              ▼
                                        Database (locked writer)  → SQLite (WAL)
```

Key facts:

- **The job runs in the daemon, not the terminal.** Ctrl-C in the CLI detaches
  the view (`stream.py`); the job keeps running. Killing the terminal doesn't
  touch it.
- **One mutating job at a time, globally.** A second submit while one runs is
  rejected with `BusyError` → HTTP 409 → `BusyResponse` on the client.
- **Progress is SSE**, pushed from the worker via `JobContext`. Read-only
  snapshots (`status`, `roots`) are plain request/response over a separate
  read-only DB connection (`queries.py`), so they never block on the writer.
- **Fixed port 51789, one daemon per machine.** The bind is the single-instance
  lock. (Consequence: isolated-home dev/test runs collide with a live daemon on
  that port — see Gotchas.)

## 3. The part you'll extend most: adding the `scan` job

Copy the shape of `jobs/demo.py`. A job type is a `JobSpec` registered at import:

```python
# jobs/scan.py
from .context import JobContext
from .registry import JobSpec, register_job

def _run_scan(ctx: JobContext) -> None:
    root_id = ctx.params["root_id"]
    cfg = ctx.config                 # frozen Config snapshot taken at job start (§9.2)
    ctx.set_total(n_files)           # drives the progress bar
    for f in worklist:
        ctx.check_cancelled()        # cooperative cancel checkpoint (§9)
        ... hash / metadata / persist ...
        with ctx.db.transaction() as conn:   # atomic per-file write (§8 A2 step 9)
            conn.execute(...); conn.execute(...)
        ctx.progress(message=f.name) # +1 and push an SSE event

register_job(JobSpec(
    type="scan",
    handler=_run_scan,
    mutating=True,
    owned_root=lambda params: params.get("root_id"),   # per-root exclusivity (§3)
))
```

Then **import the module for its side effect** so it registers. M0 does this in
`server.py` (`from ..jobs import demo as _demo`). Add `scan` the same way (or
create a `jobs/_register.py` that imports all job modules and import *that* in
`server.py`).

### `JobContext` API (jobs/context.py) — everything a handler needs

| Member | Use |
|---|---|
| `ctx.params` | the dict submitted with the job |
| `ctx.config` | frozen `Config` snapshot (reloaded per job start, §9.2) |
| `ctx.db` | the `Database` (locked writer) — see below |
| `ctx.set_total(n)` | set the denominator for the progress bar |
| `ctx.progress(done=None, message=..., eta_s=...)` | report progress (defaults to +1) |
| `ctx.log(msg)` | emit a log line to attached clients |
| `ctx.check_cancelled()` | raise `CancelledError` if cancel requested — call at your checkpoints |
| `ctx.cancelled` | cheap boolean, same signal, non-raising |

`jobs.done`/`jobs.total` are a **progress-display counter only** (§4). They are
**not** your resume key — scan resumes via the fast-path (path+size+mtime skip)
reading committed `file_instances` rows. Don't lean on `done` for correctness.

### Writing to the DB from a job

Use `ctx.db` (a `Database` in `db/connection.py`), the daemon's **single write
connection guarded by a lock** — the API thread and worker thread share it safely.

- `ctx.db.execute(sql, params)` — one autocommitted statement.
- `ctx.db.query(sql, params)` / `query_one(...)` — reads.
- `with ctx.db.transaction() as conn:` — **atomic multi-row unit**. Use this for
  the "single transaction" writes the plan requires (e.g. §8 A2 step 9: asset +
  file_instance + phash together, all-or-nothing).
- For **read-only snapshots** unrelated to the job, prefer `queries.py`'s
  read-only connection so you don't contend with the writer.

`Database._safe_write` already swallows "closed database" errors during shutdown
— a still-running worker writing as the daemon exits is harmless (reconciliation
fixes state on next start). Keep that contract: don't add raw writes that assume
the connection is always open.

## 4. Per-root exclusivity — what the queue enforces vs. what M1 must add

PLAN.md §3 guarantee 2: *a root has at most one active operation.* The queue
(`queue.py::_root_holder`) already checks, for a job whose `owned_root` returns a
`root_id`, whether that root has a **pending `review_runs`** or an **open
`merge_runs`** — and rejects with a "root busy" `BusyError` if so. So setting
`owned_root` on the scan `JobSpec` gives you the §8 A2 step 1a rejection for free.

**But the queue reject is binary.** PLAN.md distinguishes:
- **manual `scan <root>`** → error, name the holder. (The queue reject does this.)
- **`--all` / scheduled scan** → **skip that root and log it**, don't fail the
  sweep.

So for `scan --all`, M1 should **check ownership per-root before enqueuing each**
(read `review_runs`/`merge_runs`, or expose a helper) and skip+log the busy ones,
rather than submitting and catching a hard reject. The single-job global lock also
means `--all` is really "scan roots one at a time"; decide whether that's one job
iterating roots or N submitted jobs (one job iterating is simpler and matches the
single-worker model).

Reconciliation note: scan needs **no special resume** — a re-run continues via the
fast-path. `reconcile.py` already flips an orphaned `running` scan to
`interrupted`; the user just re-runs. There's a **hook in `reconcile.py`** for the
dedup/cleanup analyze-rollback (M3), currently a no-op comment — ignore it for M1.

## 5. Config knobs M1 uses (§9.2)

`config.py` exposes a frozen `Config`. The pieces M1 touches:

- `cfg.allowlist.media_exts()` → the set of extensions that qualify as media
  (photo+video, plus RAW iff `allowlist.raw`). Use this to build the scan
  worklist. Extensions are lower-case, no dot.
- `cfg.allowlist.raw` — whether the RAW group is included.
- `cfg.fastpath.mtime_tolerance_s` — the tolerant-mtime skip window (§8 A2 step 4;
  `0` = strict path+size+mtime).
- `cfg.smb.scan_workers` — concurrency cap for hashing/decoding streams (§10.1).

Config is **read per job start** and snapshotted onto `ctx.config`. Don't re-read
inside a job. `PHOTO_EXTS` / `VIDEO_EXTS` / `RAW_EXTS` constants also live in
`config.py` — `_classify`-style media-type-by-extension logic should reuse them.

The per-root `--ignore` globs are **not** in config; they belong on the `roots`
row (`roots.ignore_globs`, stored at register time). M1 must implement the glob
matching for the walker (§8 A1).

## 6. Tables M1 writes (all already defined in `db/schema.py`)

M1 is the first code to populate the catalog. Relevant tables:

- **`roots`** — `roots register` inserts (path unique; `name` unique NOCASE;
  `kind` library/trash; `ignore_globs` JSON; `last_full_scan_at`). Overlap +
  unique-leaf-name checks are M1 logic (§8 A1), not DB constraints.
- **`assets`** — one row per content hash (`content_hash` UNIQUE). Set
  `media_type` by **extension** (not by decoding), `status='active'`,
  `undecodable`/`decode_error` per §8 A2 step 8. (Perceptual columns via
  `phash`/`vphash` are **M2**, not M1.)
- **`file_instances`** — presence = row existence; **UNIQUE(root_id, path)**.
  Upsert on that key (§8 A2 step 6). Store `path` in the canonical long-path-safe
  form.
- **`jobs`** — the queue manages this; you don't write it directly.

M1 does **not** write `phash`/`vphash` (M2), `similarity_edges`/`review_*` (M3),
`merge_*` (M5), or `embeddings` (M7). Deletion-detection (§8 A2 step 11) deletes
`file_instances` and cascades forgotten assets — rely on `ON DELETE CASCADE`
(already set; `foreign_keys=ON` is enabled in `connect()`).

**Watch the fast-path predicate** (§8 A2 step 4 "fully fingerprinted"): in M1 there
are no perceptual rows yet, so define the predicate carefully now so M2 slots in
without a rescan. The schema's `assets.undecodable` / `decode_error` comments spell
out the intended semantics.

## 7. Adding CLI commands (cli/main.py)

Commands are Typer functions on `app`. They should be **thin**: resolve args,
`ensure_daemon()`, submit, stream. See `demo` / `status` / `roots` for the pattern.

- `_client_or_spawn()` returns a live `DaemonClient` (auto-spawns).
- Job-submitting commands stream with `stream.py::stream_job(client, job_id)`,
  which implements Ctrl-C-detaches. Support `--detach` (submit + return).
- Catch `BusyResponse` and print it (see `_print_busy`).
- `status` and `roots` already exist and read from the daemon snapshots; M1 will
  make them show real data once the catalog is populated. Extend
  `queries.status_snapshot()` / `roots_snapshot()` as needed (they're the single
  read surface, §11).

**Root argument resolution (§11 path-vs-`--name`) does not exist yet** — M1 must
add it: canonicalize the arg as a path and match `roots.path`; else match
`roots.name` case-insensitively; else error. `untrash` is excluded (that's M4).

A `Typer` gotcha specific to this repo: request-body Pydantic models for FastAPI
must be **module-level** (see server.py `SubmitJobRequest`) because
`from __future__ import annotations` + a locally-scoped model breaks FastAPI's type
resolution. Not a CLI issue, but you'll touch server.py if you add endpoints.

## 8. Gotchas learned in M0 (don't rediscover these)

1. **Fixed port = one daemon per machine.** Running the CLI under an isolated
   `$PACKRAT_HOME` while another daemon (different home/token) is bound to 51789
   → `401 invalid or missing token`. Before isolated-home manual runs, kill the
   stray: `netstat -ano | grep :51789 | grep LISTENING` → `Stop-Process -Id <pid>
   -Force`. For tests, prefer the in-process `TestClient` (never binds the port)
   and set `$PACKRAT_HOME` per test via the `packrat_home` fixture.
2. **Windows console windows.** The daemon is spawned windowless via
   `pythonw.exe` + `CREATE_NO_WINDOW | CREATE_NEW_PROCESS_GROUP` (`spawn.py`).
   Do **not** combine `DETACHED_PROCESS` with `CREATE_NO_WINDOW` — they conflict
   and a window appears. If you spawn subprocesses (e.g. ffmpeg/exiftool in scan),
   pass `CREATE_NO_WINDOW` so they don't flash windows either.
3. **UTF-8 output.** CLI + daemon reconfigure stdio to UTF-8 (`·`, `●`, `⚠` glyphs
   mangle on legacy Windows codepages otherwise). Keep that in new entrypoints.
4. **`exiftool` is a binary, not just a Python package.** `pyexiftool` wraps
   `exiftool.exe`, which must be on PATH. It's installed on this machine
   (13.59); M1's metadata step should still degrade gracefully if a probe fails
   (metadata is best-effort per §8 A2 step 7 — never crash the file).
5. **Decode is the gate (§9.1).** All decode/hash/PDQ wheels are confirmed working
   on this Windows + CPython (blake3, pillow, pillow-heif, PyAV, pdqhash, rawpy).
   HEIC/AVIF decode via `pillow_heif.register_heif_opener()` (note: pillow-heif
   1.4 dropped `register_avif_opener` — the one call covers both). See `smoke.py`
   for working decode paths you can lift for scan's hashing/decoding.

## 10. Running & testing

```sh
uv sync --extra media --extra dev     # runtime + decode stack + pytest

uv run pytest -q                      # full suite
uv run packrat daemon status          # is it up?
uv run packrat demo                   # exercise the runtime (throwaway job)
uv run packrat smoke-test --generate  # §9.1 decode self-test (synthesizes samples)
```

Test conventions (see `tests/`):
- `packrat_home` fixture (conftest.py) isolates `$PACKRAT_HOME` per test.
- Prefer the in-process Starlette `TestClient` for API tests (`test_api.py`) —
  no port binding, no daemon process.
- For queue/job tests, use a `JobQueue` + `Database` directly and **join the
  worker before closing the DB** (`queue.shutdown()`), as `test_jobs.py` does, to
  avoid teardown races.
- The `demo` job (`jobs/demo.py`) is the reference for a cancellable,
  progress-emitting job — read it before writing `scan`.

## 11. One-paragraph mental model

The daemon is a durable single-worker executor. An "operation" is a **job type**:
a registered `JobSpec` whose handler receives a `JobContext` giving it a config
snapshot, a locked DB writer, a progress channel, and a cancel flag. The queue
enforces "one mutating job at a time" and per-root exclusivity; the DB (full §4
schema, WAL, cascades, partial-unique indexes) is the source of truth; clients
only submit/stream/cancel. M1 adds the first handler (`scan`) and the first
catalog writes — you should be adding a `jobs/scan.py`, `roots` commands, and
resolution/walker helpers, **not** touching the queue, daemon, or schema.
